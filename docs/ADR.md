# Architecture Decision Records

## 철학
**작게 시작하고, 반복이 두 번째로 보일 때만 추출한다.** 처음부터 추상화하면 잘못된 인터페이스를 박제한다. 워크스페이스·공유 패키지·디자인 토큰 모두 같은 원칙을 따른다. 동작하는 가장 단순한 구현을 먼저 두고, 두 번째 진짜 사용처가 나타날 때 형태가 보이는 만큼만 빼낸다.

부수 원칙:
- **외부 의존성 최소화.** 직접 짤 수 있고 유지비가 낮다면 라이브러리 대신 직접 짠다.
- **빌드 시점에 끝낸다.** 정적 생성으로 가능한 작업은 런타임에 미루지 않는다.
- **타입은 입력 경계에서 좁힌다.** `any`/`as unknown as`로 우회하지 않는다.

---

### ADR-001: pnpm 모노레포 (apps/* + packages/*)
**결정**: 패키지 매니저로 pnpm을, 워크스페이스 구조로 `apps/*` + `packages/*`를 채택한다.

**이유**: 첫 결과물(블로그) 이후에도 같은 작업자(본인)가 여러 프론트엔드 앱을 만들 가능성이 높다. 매번 셋업을 다시 까는 비용을 피하려면 워크스페이스가 필요하다. pnpm은 디스크 효율, 호이스팅 엄격성, 네이티브 워크스페이스 지원 면에서 npm/yarn보다 낫다.

**트레이드오프**: 모노레포라는 추가 복잡도. 단일 앱일 동안에는 약간의 오버엔지니어링이지만, 두 번째 앱이 들어올 때 회수된다.

---

### ADR-002: Next.js 16 App Router + React 19
**결정**: 프레임워크는 Next.js 16의 App Router, 라이브러리는 React 19로 간다. 페이지 라우터(`pages/`)는 사용하지 않는다.

**이유**: 블로그는 정적 생성이 중심이고, App Router의 Server Component + `generateStaticParams` 조합이 이 패턴에 가장 잘 맞는다. React 19의 Actions·`use`·서버 컴포넌트 통합으로 클라이언트 코드를 더 줄일 수 있다.

**트레이드오프**: 16은 비교적 최근 버전이라 학습 시점 기억과 깨지는 변경(캐싱·`fetch` 동작·메타데이터 API 등)이 있다. 코드를 만지기 전에 `node_modules/next/dist/docs/`의 해당 문서를 먼저 읽는 비용을 감수한다.

---

### ADR-003: Tailwind CSS v4, CSS-first 구성
**결정**: 스타일링은 Tailwind v4를 쓴다. 설정은 `@import "tailwindcss"`와 `@theme` 블록 기반의 CSS-first 방식으로만 한다. `tailwind.config.{js,ts}`는 만들지 않는다.

**이유**: v4는 CSS 안에서 토큰을 직접 정의하므로 빌드 도구와의 결합이 줄고, 토큰 변경이 즉시 보인다. 모노레포 전반의 토큰 공유 시점에도 CSS preset 단일 파일로 끝난다. JS 설정 파일을 두 번째 앱과 동기화하는 비용이 사라진다.

**트레이드오프**: v3 시절의 일부 플러그인 생태계는 v4에서 미지원이거나 다르게 동작한다. 필요한 플러그인이 v4 비호환이면 직접 CSS로 대체한다.

### ADR-004: 콘텐츠 소스는 파일 기반 MDX
**결정**: 블로그 글은 `apps/blog/content/posts/*.mdx`에 두고, frontmatter로 메타데이터를 관리한다. CMS·DB는 도입하지 않는다.

**이유**: 단일 저자, 자기 발행 블로그에서는 글이 곧 코드 변경이다. 에디터·git·PR 흐름이 발행 워크플로우가 된다. 외부 서비스 의존성과 그 비용·렌더 지연·다운타임 리스크를 모두 제거한다. MDX는 본문 안에서 React 컴포넌트를 호출할 수 있어 그래프·인터랙션 같은 글 단위 확장이 자연스럽다.

**트레이드오프**: 비기술자 협업이 어렵다. 단일 저자 전제가 깨지면 재검토 대상이다. 검색·전문 색인 같은 기능은 직접 만들어야 한다 (MVP 제외).

---

### ADR-005: TypeScript strict, 타입 우회 금지
**결정**: 모든 워크스페이스 패키지는 `strict: true`로 시작한다. `any`·`as unknown as` 같은 우회는 금지하고, 좁힐 수 없으면 입력 경계를 다시 잡는다.

**이유**: 블로그 자체는 타입 위험이 적지만, 모노레포 전체 규칙을 한곳에서 정해두어야 이후 앱들에서도 동일한 기준이 유지된다. strict를 처음부터 켜두는 비용이 나중에 끄는 비용보다 훨씬 작다.

**트레이드오프**: 외부 라이브러리 타입이 부실하면 우회를 쓰고 싶어진다. 그럴 때는 우회 대신 좁은 어댑터 함수로 격리한다.

---

### ADR-006: 다크/라이트 모드를 동시에 1급 시민으로
**결정**: 색 토큰은 라이트·다크 모두 정의한다. 시스템 설정을 따르는 것이 기본 동작이고, 사용자가 헤더에서 명시적 토글로 덮어쓸 수 있다.

**이유**: 둘 중 하나만 지원하면 토큰을 한쪽 기준으로만 다듬게 되고, 나중에 다른 쪽을 추가할 때 모든 토큰을 다시 손봐야 한다. 처음부터 페어로 정의해야 양쪽이 모두 의도된 결과물이 된다.

**트레이드오프**: 컬러 결정이 두 배가 된다. 단일 포인트 색을 양쪽 모드에서 모두 검증해야 한다.
